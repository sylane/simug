# Task 5.4: Orchestrator-owned issue analysis comments

## Objective
- Post deterministic orchestrator-owned analysis comments on triaged issues based on validated `issue_report` actions.

## Scope
- In: GitHub issue comment mutation path, triage-to-comment formatting, deterministic marker metadata, integration/unit tests.
- Out: planning insertion, task bootstrap from inserted tasks, PR backlinking.

## Key Decisions
- Marker format: deterministic HTML marker `<!-- simug:issue-triage:v1 issue=<n> relevant=<bool> needs_task=<bool> -->` included in every orchestrator triage comment.
- Idempotency policy: before posting, scan issue comments and skip posting only when an existing marker comment from the authenticated worker user is already present.
- Comment shape: machine-auditable, deterministic body including marker, issue metadata, analysis text, and optional proposed task metadata.

## Validation
- Baseline: `GOCACHE=/tmp/go-build go test ./...` -> pass.
- Focused: `GOCACHE=/tmp/go-build go test ./internal/app ./internal/github` -> pass.
- Final: `GOCACHE=/tmp/go-build go test ./...` -> pass.
- Coverage: `GOCACHE=/tmp/go-build GOEXPERIMENT=nocoverageredesign go test ./... -coverprofile=coverage.out && GOCACHE=/tmp/go-build go tool cover -func=coverage.out` -> pass (`total: 64.0%`).

## Residual Risks
- Duplicate suppression currently depends on marker presence in comments by the authenticated worker user; if the marker format changes in future versions, migration compatibility must be handled explicitly.
