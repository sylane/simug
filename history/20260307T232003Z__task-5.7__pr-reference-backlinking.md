# Task 5.7: PR reference back-linking to issue

## Objective
- Post deterministic issue comments linking the created implementation PR and inserted task ID after issue-derived bootstrap creates a managed PR.

## Scope
- In: issue backlink comment formatting/posting, idempotency marker checks, orchestration wiring and tests.
- Out: full issue-first e2e matrix and failure injection coverage.

## Key Decisions
- Backlink context: emit issue-to-PR backlink only when both `active_issue` and `pending_task_id` are set at PR creation time.
- Marker model: use deterministic backlink marker (`simug:issue-pr-link:v1 issue=<n> task=<id> pr=<n>`) and skip duplicates only for comments authored by the authenticated worker user.
- Wiring point: post backlink immediately after successful PR creation and before clearing issue/task state.

- Baseline: `GOCACHE=/tmp/go-build go test ./...` -> pass.
- Focused: `GOCACHE=/tmp/go-build go test ./internal/app` -> pass.
- Final: `GOCACHE=/tmp/go-build go test ./...` -> pass.
- Coverage: `GOCACHE=/tmp/go-build GOEXPERIMENT=nocoverageredesign go test ./... -coverprofile=coverage.out && GOCACHE=/tmp/go-build go tool cover -func=coverage.out` -> pass (`total: 65.8%`).

## Residual Risks
- Backlink dedupe marker includes PR number; intentional second PRs for the same issue/task will create distinct backlink comments, which is useful for traceability but can increase issue-thread verbosity.

## Planning Refinements
- Refined Task 5.8 scope to explicitly include PR-backlink idempotency coverage.
