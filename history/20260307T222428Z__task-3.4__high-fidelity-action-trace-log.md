# Task 3.4: High-fidelity action trace log

## Objective
- Add structured, high-fidelity trace logging so failed runs can be reconstructed offline.

## Scope
- In: command/action tracing for orchestrator decisions and git/gh invocations.
- Out: new loop modes or issue-triage behavior changes.

## Key Decisions
- Use `.simug/events.log` as the single trace artifact and add richer event kinds instead of introducing a second log file.
- Add command trace hooks in `internal/git` and `internal/github` so every command path emits structured trace metadata (duration, exit code, stdout/stderr tails, error text).
- Add `run_id` and `tick_seq` correlation fields to orchestrator events and command traces, plus `command_seq` for per-run command ordering.
- Emit explicit `invariant_decision` events (pass/fail) for branch-policy, checkout synchronization, post-agent validation, and main-readiness checks.

## Validation
- Baseline: `GOCACHE=/tmp/go-build go test ./...` -> Pass
- Baseline: `GOCACHE=/tmp/go-build GOEXPERIMENT=nocoverageredesign go test ./... -coverprofile=coverage.out` -> Pass
- Baseline: `GOCACHE=/tmp/go-build go tool cover -func=coverage.out` -> Pass
- Final: `GOCACHE=/tmp/go-build go test ./...` -> Pass
- Final: `GOCACHE=/tmp/go-build GOEXPERIMENT=nocoverageredesign go test ./... -coverprofile=coverage.out` -> Pass
- Final: `GOCACHE=/tmp/go-build go tool cover -func=coverage.out` -> Pass

## Residual Risks
- Command traces from mocked runners in tests cannot always provide true process-level stderr/exit metadata; runtime `os` command paths do capture these fields.

## Planning Refinements
- Updated `Task 3.5` scope to require archival correlation with `run_id`, `tick_seq`, and `command_seq` lineage.
