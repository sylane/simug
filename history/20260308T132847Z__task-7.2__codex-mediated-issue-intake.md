# Task 7.2: Replace planner insertion with Codex-mediated issue-task intake

## Objective
- Complete the post-7.1 issue-derived bootstrap handoff so `issue_report.needs_task=true` yields deterministic Codex-targeting context without orchestrator planning-file mutation.

## Scope
- In: issue-triage state handoff, bootstrap intent prompt/validation, issue backlink and issue-update task-context semantics, and related tests/docs.
- Out: protocol schema overhauls beyond task handoff requirements and session-runtime redesign work tracked in later phases.

## Key Decisions
- Added explicit issue-derived handoff persistence via `state.issue_task_intent` (`issue_number`, `task_title`, `task_body`, `recorded_at`) so triage output survives into bootstrap intent planning without orchestrator planning-file edits.
- Kept protocol schema stable (`INTENT_JSON` unchanged) and injected issue-task proposal context through bootstrap intent prompt instructions; this avoids a protocol contract change while still making issue-driven intent deterministic.
- Derived legacy-compatible `pending_task_id` from approved `bootstrap_intent.task_ref` and introduced `state.active_task_ref` so issue backlink and issue-update comments can carry validated task context.
- Moved post-PR state cleanup (`pending_task_id`, `issue_task_intent`, `bootstrap_intent`, `bootstrap_session_id`) to after action/comment application so same-tick orchestrator-owned issue updates can still include task context.
